# 2026-01-10 — Release 4.4.32 postmortem (process + pitfalls)

## Context

Release `4.4.32` shipped fixes for repeated Tradier polling events and unbounded live memory growth, plus runtime
telemetry and broker smoke apitests. The release process was messy enough that we want a short “what happened / what to
do next time” note.

## What went well

- `4.4.32` is published on PyPI.
- The OOM-mitigation fixes are covered by deterministic regression tests, including “large Tradier order history”
  simulations.
- Added opt-in broker API smoke tests (`pytest -m apitest`) for Alpaca (paper) and Tradier (mixed; see below).

## What was messy / confusing

- **Tag drift**: after publishing to PyPI, the git tag `v4.4.32` was not present locally (latest seen tag was `v4.4.31`).
  The repo’s release workflow is tag-driven (`.github/workflows/release.yml`), so pushing the tag *after* a manual PyPI
  publish will usually cause the publish job to fail (PyPI rejects duplicate version uploads) and may prevent the GitHub
  Release from being created automatically.

## Concrete lessons

- **Preferred path**: tag first → let the GitHub Action publish to PyPI + create the GitHub Release.
- If PyPI was already published manually:
  - Avoid “fixing” it by pushing `vX.Y.Z` afterward unless you’re okay with a failed publish workflow.
  - Better: create a dedicated “GitHub Release only” workflow for backfills, or manually create the GitHub Release
    (future improvement).

## Broker apitest lesson (Tradier sandbox)

- Tradier “test” creds (`TRADIER_TEST_*`) hit the Tradier sandbox (`sandbox.tradier.com`). Some order lifecycle endpoints
  (notably cancel) can behave differently than live and may reject ids returned by sandbox order placement.
- Therefore:
  - Keep “connectivity/balances/orders list” smoke tests on sandbox.
  - Run “submit + cancel lifecycle” against live Tradier only when intentionally configured.

## Next steps (post-release hygiene)

- Start the next version branch from `dev`: `version/4.4.33`.
- Add a placeholder changelog header for `4.4.33` (“Unreleased”) and continue work there.
- Ensure bot_manager deploy uses `LUMIBOT_VERSION=4.4.32` (repo variable) so new images pick up the latest LumiBot.
